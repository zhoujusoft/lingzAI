## 3. 开场 1 分钟

### 3.1 可以直接说的话

我们这次不是在做一个“通用 AI 平台”，而是在给当前项目的 Python 技能执行补一层真正可落地的沙箱运行时。  
目标很明确：把技能执行从主后端进程里拆出来，把版本、文件、日志、产物和会话关系稳定下来，先把边界做对，再考虑后续规模化。

### 3.2 一句话结论

本轮推荐方案是：

**平台基础镜像 + 技能版本独立 venv + 会话级 Docker 沙箱实例**

---

## 4. 背景与问题 1 分钟

### 4.1 当前问题

现在的技能执行有 5 个核心问题：

1. 技能代码和主后端在同一信任边界
2. Python 依赖容易污染和冲突
3. 文件访问边界不清晰
4. 会话、版本、执行、产物之间没有稳定绑定
5. 失败排障和后台治理能力不够

### 4.2 为什么现在就得做

如果后面继续增加代码执行型技能，而不先做沙箱层，这些问题会越来越难收拾：

1. 风险会从“局部问题”变成“系统性问题”
2. 后续再补版本和文件语义会更痛
3. 运维、排障和后台治理会越来越乱

---

## 5. 方案选型 2 分钟

### 5.1 为什么不用用户级长驻沙箱

用户级沙箱的问题是：

1. 多会话状态容易互相污染
2. 审计和问题定位更难
3. 资源回收难，企业场景更容易失控

所以我们默认不选用户级。

### 5.2 为什么选会话级沙箱

会话级沙箱的优点：

1. 边界自然，符合聊天产品心智
2. 状态污染面最小
3. 最容易做审计、回放和版本治理

### 5.3 为什么先选 Docker，不直接上 Kubernetes

这轮重点不是做平台编排，而是先把执行模型设计正确：

1. Docker 足够支撑 MVP
2. 团队当前对 Kubernetes 不熟
3. 过早上 Kubernetes 会把重点带偏到平台复杂度

所以结论是：

1. MVP 先用 Docker
2. Kubernetes 作为后续演进方向

## 6. 核心架构 3 分钟

### 6.1 先讲最关键的一句话

**技能是打包和版本维度，不是运行实例维度。**

也就是说：

1. 技能版本对应一份安装产物
2. 真正运行时是按会话启动沙箱实例

### 6.2 运行环境模型

这次不采用“一个技能一个完整业务镜像”，而是采用四层：

1. 平台基础镜像
2. 技能目录
3. 技能版本独立 venv
4. 会话工作区

这样做的好处：

1. 技能间依赖不冲突
2. 不需要维护一堆完整业务镜像
3. 后续更容易演进到 Kubernetes

### 6.3 会话与版本模型

这次明确两个关键规则：

1. 一个 `SKILL_CHAT` 会话固定绑定一个 `skillId`
2. 会话首次执行时绑定一个具体 `install_id`

绑定之后：

1. 不自动跟随新版本
2. 旧版本下线后，会话转 `READ_ONLY`
3. 要体验新版本，就新建会话

### 6.4 文件与执行协议

文件这块也要讲清楚三个点：

1. 输入不是传真实路径，而是传稳定快照引用
2. 平台执行前统一投影到 `run/inputs`
3. 技能结果必须写 `result.json`

核心原则：

1. `input.json` 是唯一输入协议
2. `result.json` 是唯一主结果协议
3. `stdout/stderr` 只是日志，不是主结果

### 6.5 共享文件模型

`session_shared` 不是默认用户可见目录，只是跨轮复用区。

只有两件事同时满足，文件才进入正式引用空间：

1. 本次执行成功
2. 文件在 `result.json` 中被显式声明

## 7. 这轮边界与不做项 2 分钟

### 7.1 这轮必须做

1. Python 技能执行沙箱化
2. 会话级工作区
3. 版本绑定
4. 文件快照模型
5. 固定执行协议
6. 执行记录与日志治理

### 7.2 这轮明确不做

1. Kubernetes 落地
2. 多语言运行时
3. 系统级依赖安装
4. 用户级长驻沙箱
5. 在线进入容器调试
6. 外部技能无适配直接接入

### 7.3 资源约束

默认约束已经收敛好了：

1. 单次执行超时 `60 秒`
2. 单实例 `1 CPU + 1GB`
3. 单会话工作区 `500MB`
4. 同会话同一时刻只允许一个执行
5. 沙箱默认完全禁网

## 8. 评审会需要拍板的 5 个点 2 分钟

这部分建议你最后明确抛出来，方便大家直接给结论。

### 8.1 拍板项

1. 是否确认 **Docker + 会话级沙箱** 作为 MVP 主路线
2. 是否确认 **平台基础镜像 + 技能目录 + 独立 venv**，而不是“一个技能一个完整镜像”
3. 是否确认 **首次执行绑定 install_id，旧会话不自动升级版本**
4. 是否确认 **文件引用绑定快照，不跟随最新版本漂移**
5. 是否确认 **失败不提交共享变更，但保留最小必要失败现场**

### 8.2 拍板后的直接结果

如果这 5 个点通过，后面就可以直接进入：

1. 数据模型落库
2. 技能导入链路改造
3. Docker 执行主链路打通
4. 后台治理能力补齐

## 9. 结尾 30 秒

### 9.1 收口话术

这次方案的重点不是把平台一步做大，而是先把技能执行这条链路的边界、版本、文件和治理语义做稳定。  
只要本轮把执行模型定住，后面无论是继续留在 Docker，还是往 Kubernetes 演进，整体抽象都不会推倒重来。
