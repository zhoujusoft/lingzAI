# 技能沙箱设计讨论纪要

## 背景

本纪要用于记录围绕“技能执行沙箱化”的一轮设计讨论，目标是为当前项目引入可落地的 Python 技能沙箱执行方案。当前项目面向企业私有部署，不考虑 SaaS 多租户公有云版本。

讨论重点包括：

* 技能沙箱应该如何构建
* 技能与沙箱实例之间是什么关系
* Python 运行时应该如何隔离
* 会话级 sandbox 与用户级 sandbox 哪种更合适
* Docker 与 Kubernetes 分别适合放在什么阶段
* AgentScope 的模式有哪些可借鉴点

## 当前项目现状

结合现有代码，当前技能执行能力还没有真正进入独立沙箱，主要特征如下：

1. 技能聊天链路已经具备“技能感知”和“工具动态注入”能力。
2. Python 执行目前仍由后端进程内的 helper 通过 `ProcessBuilder` 直接拉起脚本执行。
3. 文件读写仍然可以触达宿主挂载目录。
4. 技能导入时会直接在当前运行环境中执行 `pip install -r requirements.txt`。
5. 部署层面通过 Docker Compose 把 `skills`、`uploads`、`python-runtime` 以 volume 形式挂进后端容器，共享边界较大。

当前直接风险：

* 技能执行与主后端服务进程没有强隔离
* Python 依赖可能互相污染
* 技能脚本和临时文件缺少独立边界
* 后续如果引入更多代码执行技能，运维和安全风险会快速上升

## 讨论结论一：技能是打包维度，不是运行实例维度

在讨论中先明确了一个核心概念：

* `技能` 更适合作为打包、版本、镜像或运行环境工件的维度
* `沙箱实例` 更适合作为具体执行单元的维度

也就是说：

* 一个技能版本对应一份受控执行环境
* 但真正运行时，不应该默认“一个技能长期共用一个实例”
* 更合理的做法是：按会话或任务创建具体的运行实例

这样做的好处是：

* 技能版本边界清晰
* 执行状态不会天然跨上下文污染
* 更容易做审计、回放和问题定位

## 讨论结论二：优先采用会话级 sandbox

围绕“会话级 sandbox”与“用户级 sandbox”做了比较。

### 会话级 sandbox

定义：

* 同一用户在同一技能会话中复用一个沙箱实例
* 会话结束或超时后回收实例

优点：

* 隔离边界自然，符合聊天产品语义
* 状态污染范围最小
* 更容易做日志、回放、审计和故障定位
* 资源核算更清楚

缺点：

* 冷启动频率高
* 大缓存或长任务状态复用能力较弱

### 用户级 sandbox

定义：

* 同一用户在同一技能下长期复用一个沙箱实例
* 多个会话只是逻辑区分，底层执行环境是共享的

优点：

* 热启动快
* 用户级缓存利用率更高
* 对连续调试型技能更友好

缺点：

* 会话之间容易相互污染
* 资源回收和失控治理更复杂
* 企业级审计、定位、权限边界都更难处理

### 本轮推荐

默认主模型采用：

* `技能级打包/镜像`
* `会话级 Docker 沙箱实例`

用户级 sandbox 不作为默认方案，只保留为未来少数特殊技能的增强选项。

## 讨论结论三：借鉴 AgentScope，但不照搬

讨论中明确提出希望借鉴 AgentScope 的模式。

可借鉴点：

* 把 sandbox 作为独立运行时能力，而不是把代码执行继续内嵌在主应用进程
* 支持基于 `session_id` 复用会话内实例
* 将执行环境、生命周期和主 Agent 流程解耦

不直接照搬的原因：

* 当前项目已有自己的技能包、技能管理后台和工具注入体系
* 当前目标是企业私有部署，不是通用 Agent Runtime 平台
* 我们更需要的是“适配现有技能体系的运行时层”，而不是整体替换为另一套框架

因此最终选择的方向是：

* 学习 AgentScope 的 session sandbox 思路
* 但运行时抽象、技能包集成和管理入口仍然围绕本项目现有结构设计

## 讨论结论四：Docker 和 Kubernetes 的定位不同

在讨论中也梳理了 Docker 与 Kubernetes 的区别。

### Docker 更适合当前阶段

Docker 更像是“在单机或少量节点上启动和管理容器”的执行引擎。

在本项目里的作用可以理解为：

* 主后端发起技能执行请求
* 通过 Docker 启动一个独立 Python 沙箱容器
* 挂载技能包目录、只读资源目录和临时工作目录
* 执行脚本或入口命令
* 收集 stdout/stderr 和结构化事件
* 执行结束或超时后回收容器

优点：

* 学习曲线低
* MVP 能快速落地
* 非常适合先把“安全执行边界”做对

### Kubernetes 更适合作为后续演进

Kubernetes 更像是“在多机环境中调度和管理大量容器”的平台。

它更适合解决的问题是：

* 多节点调度
* 高并发执行
* 故障自动迁移
* 更细粒度的资源配额
* 与 RuntimeClass / gVisor / Kata 等更强隔离运行时结合

缺点：

* 学习和运维成本明显更高
* 对当前阶段来说，容易把重点从“技能沙箱模型”转移到“平台建设”

### 本轮决策

由于当前团队对 Kubernetes 经验有限，且本轮主要目标是先把技能沙箱模型设计清楚，因此决定：

* `MVP 先使用 Docker 模式`
* `Kubernetes 作为后续生产级演进方向`

也就是说，本轮设计文档应优先围绕 Docker 方案展开，再保留未来向 Kubernetes 升级的抽象接口和演进路线。

## 推荐的 MVP 方案

### 目标模型

* 技能作为打包和版本维度
* 每个技能版本对应一个受控运行环境
* 每个技能会话分配一个独立 Docker 沙箱实例
* 会话内可复用，跨会话默认不共享

### Python 沙箱的基础要求

* 独立容器执行，不直接在主后端进程内跑 subprocess
* 容器内固定 Python 版本
* `requirements.txt` 在沙箱镜像或受控初始化阶段安装
* 技能目录只读挂载
* 工作目录使用独立临时卷
* 默认关闭外网，按技能白名单或企业策略开放
* 强制 CPU / 内存 / 执行时长限制
* 保留 stdout / stderr / exit code / 结构化日志

### 需要补齐的系统能力

* `SandboxManager`：负责创建、查询、销毁沙箱
* `SandboxImageBuilder`：负责技能运行环境构建
* `SandboxSessionAllocator`：按技能 + 会话分配实例
* `SandboxExecutionGateway`：为聊天工具调用提供统一执行入口
* `SandboxAuditLog`：记录执行审计信息

## 为什么不建议继续沿用当前模式

当前模式虽然实现成本低，但它本质上只是“共享 Python 运行时 + 进程级执行”，不能视为真正的企业级沙箱方案。

主要原因：

* 无法形成独立信任边界
* 不利于控制依赖污染
* 不利于限制技能对宿主资源的影响
* 难以支撑后续更多代码执行技能

因此，当前模式最多只能作为过渡或开发态方案，不适合作为正式的企业级技能执行模型。

## 本轮最终结论

本轮讨论最终收敛为以下决策：

1. 技能沙箱需要独立运行时层，不能继续把 Python 执行内嵌在主后端进程里。
2. 技能应作为打包/版本维度，沙箱实例应作为执行维度。
3. 默认采用会话级 sandbox，不默认采用用户级长驻 sandbox。
4. 借鉴 AgentScope 的 session sandbox 思路，但不照搬其整套平台结构。
5. MVP 先采用 Docker 模式实现。
6. Kubernetes 作为后续企业级规模化演进方向。
